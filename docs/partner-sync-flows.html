Dad Signs Up & Invites Mom
Account creation, onboarding, family setup, and invite link generation
User Action
Dad visits thedadcenter.com
Lands on marketing page. Clicks "Get Started" CTA.
Signup
/signup — Create Account
Email + password or Google OAuth via signInWithGoogle(). Zod-validated form: name (2+ chars), email, password (8+ chars), confirm.
System
Email Confirmation
Supabase sends confirmation email. User clicks link.
Redirect
/auth/callback → /onboarding
Auth callback exchanges code for session, then redirects to onboarding welcome page.
Onboarding Step 1 of 3
/onboarding/role — Select Role
Dad picks "Dad" (or Mom / Other). Auto-advances on tap — no Continue button. Calls useUpdateRole() mutation.
Onboarding Step 2 of 3
/onboarding/family — Family Setup
Toggle: Expecting vs Baby Born. Sets due date or birth date + optional baby name.
▶Calls create_family_for_user() RPC — creates family + generates invite_code (12-char hex)
▶Sets signup_week on profile for free window calculation
▶Calls initialize_family_tasks_with_catchup() — generates ~200 personalized tasks
▶Shows animated progress bar during task generation
Onboarding Step 3 of 3
/onboarding/ready — "You're all set!"
Confirmation screen showing task count. Navigates to dashboard.
Complete
/dashboard
Priority-ordered cards. Dad sees all cards including InvitePartnerCard.
Later, on Dashboard
Invite Partner
InvitePartnerCard — Copy Invite Link
Displays 12-char invite code in monospace with copy button. "Copy Invite Link" button creates thedadcenter.com/signup?invite=ABCDEF123456 and copies to clipboard.
▶Fires trackActivity(profile.id, 'partner_invited')
▶Shows toast: "Invite link copied!"
▶Also available via Settings > Family > copy code / share link
2
Mom Receives Invite & Joins
Deep link handling, invite code lookup, atomic family join with subscription sync
Deep Link
Mom clicks invite link
/signup?invite=ABCDEF123456 — invite code is captured from URL query param and saved to localStorage as tdc_invite_code.
Signup
/signup — Create Account
Mom signs up normally. The invite code persists in localStorage through the signup + email confirmation flow.
System
Email Confirmation → /auth/callback → /onboarding
Standard auth flow. On the welcome page, useEffect checks localStorage for tdc_invite_code — if found, auto-redirects to /onboarding/role?next=join.
Onboarding Step 1
/onboarding/role?next=join — Select Role
Mom picks "Mom". The next=join query param routes to /onboarding/join instead of /onboarding/family.
Validation
/onboarding/join — Invite Code Pre-filled
Code pre-filled from localStorage. As user types (or on load), debounced validation calls lookup_family_by_invite(p_code).
▶lookup_family_by_invite() — SECURITY DEFINER, returns family_name + family_stage only
▶Shows: "Found family! Stage: Expecting"
▶Join button enabled only when family is found
Atomic Join
Clicks "Join Family" → join_family(p_invite_code)
Single SECURITY DEFINER RPC performs all operations atomically:
✓Validates invite code against families.invite_code
✓Checks caller isn't already in a family
✓Enforces COUNT(*) < 2 member limit
✓Sets family_id on Mom's profile
★Syncs subscription_tier + subscription_expires_at from existing family member
✓Sets active_baby_id to family's primary baby
→Returns { family_id, tier, baby_id }
Notification
POST /api/partner-joined — Email Dad
Fire-and-forget. Route authenticates caller, finds family owner via families.owner_id, triggers email via triggerEmail('partner_joined', owner_id, { partner_name }).
Cleanup
Clear localStorage → /onboarding/complete
Removes tdc_invite_code from localStorage. Navigates to completion screen.
Complete
/dashboard — Shared family view
Mom sees shared tasks, partner activity card. Dad's InvitePartnerCard disappears (hasPartner now true).

Subscription Propagation
How one subscription covers the whole family across 4 scenarios
⚠
All Stripe webhooks go through a single canonical handler: apps/web/src/app/api/stripe/webhook/route.ts. Idempotency is enforced via stripe_webhook_events table with UNIQUE constraint on event_id.
Webhook
customer.subscription.created
Stripe fires webhook when Dad subscribes ($4.99/mo or $39.99/yr).
Handler
handleSubscriptionUpdate()
Finds user by stripe_customer_id in subscriptions table. Determines tier from status (active/trialing = premium).
✓Updates subscriptions record (tier, status, period dates)
✓Updates Dad's profiles.subscription_tier = 'premium'
∅Propagates to family members — no-op, Mom hasn't joined yet
✉Triggers subscription_confirmed email to Dad
Later: Mom Joins
join_family() syncs subscription
The RPC queries existing family members for premium/lifetime tier and copies it to Mom's profile. Mom gets premium immediately on join.
State
Mom already joined family (both on free tier)
Both Dad and Mom are in the same family. Both have subscription_tier = 'free'.
Webhook
customer.subscription.created
Dad subscribes after Mom is already in the family.
Handler
handleSubscriptionUpdate()
Same handler as Scenario A, but this time the family propagation query finds Mom:
✓Updates Dad: subscription_tier = 'premium'
★Propagates to Mom: UPDATE profiles SET subscription_tier='premium', subscription_expires_at=... WHERE family_id=... AND id != dad_id
Result
Both users immediately premium
Mom's next page load shows premium features. No action required from her.
Webhook
customer.subscription.deleted
Stripe fires when subscription is fully canceled (not just "cancel at period end").
Handler
handleSubscriptionCanceled()
Calculates grace period: max(current_period_end, 7 days from now).
!Updates subscription status to 'canceled'
★Keeps tier as 'premium' during grace period (client checks still pass)
✓Sets subscription_expires_at on ALL family members
✉Triggers subscription_expiring email with days remaining
Grace Period
Both stay premium until subscription_expires_at
Client-side isPremium checks continue to pass. A cron job or on-read check downgrades tier to 'free' after expiry.
Post-Expiry
Both downgrade to free
Paywall overlays appear. Tasks limited to 30-day rolling window. Briefings limited to 4 weeks from signup.
Webhook
checkout.session.completed (mode: payment)
Lifetime purchase uses Stripe payment mode, not subscription mode. Metadata: plan: 'lifetime'.
Handler
handleCheckoutComplete()
Detects lifetime purchase from session metadata:
✓Upserts subscriptions: tier = 'lifetime', status = 'active'
✓Updates Dad: subscription_tier = 'lifetime', subscription_expires_at = null
★Propagates lifetime to ALL family members: WHERE family_id=... AND id != dad_id
✉Triggers subscription_confirmed email (plan: 'lifetime')
Result
Lifetime never expires
subscription_expires_at = null — no grace period, no cron downgrade, permanent premium for the whole family.

Security Layer
RLS policies, SECURITY DEFINER RPCs, subscription field protection, member limits
🔒
All security functions defined in: supabase/migrations/20260330000001_partner_sync_security_fixes.sql. Every SECURITY DEFINER function uses SET search_path = '' and fully-qualified table names.

RLS Protection

✕
families SELECT — requires is_family_member(id). Non-members cannot read any family data.
⚡
Bypassed by lookup_family_by_invite() — SECURITY DEFINER, returns only name + stage (minimal exposure).
⚡
Bypassed by join_family() — SECURITY DEFINER, performs atomic join with full validation.
✓
profiles UPDATE — auth.uid() = id (user can update own profile only).

Subscription Field Trigger

✕
current_user = 'authenticated' (client SDK) — subscription_tier and subscription_expires_at silently reverted to OLD values. Cannot modify.
✓
current_user = 'postgres' (SECURITY DEFINER) — join_family() can sync subscription tier on join.
✓
current_user = 'service_role' (admin key) — Stripe webhook handler can update subscription fields.

Who Can Modify Subscription

✓
join_family() RPC — SECURITY DEFINER runs as postgres
✓
Stripe webhook — uses getSupabaseAdmin() (service_role key)
✕
Client SDK supabase.from('profiles').update() — trigger silently blocks changes

Member Limit

✓
join_family() checks COUNT(*) FROM profiles WHERE family_id = v_family_id
✕
If count ≥ 2: RAISE EXCEPTION 'This family already has the maximum number of members'
✕
If caller already in a family: RAISE EXCEPTION 'You are already in a family'
✓
Invite code regeneration: only families.owner_id can call regenerate_invite_code()
