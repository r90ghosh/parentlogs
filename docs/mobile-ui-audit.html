Executive Summary

Audit Overview

The mobile app has a strong foundation — good component architecture, consistent brand identity, and working theme infrastructure. However, there are systemic issues that prevent it from feeling like a polished iOS-native app.

Critical No Centralized Color Tokens

1,494 hardcoded color values scattered across 103 files. Changing a single brand color requires editing every file manually. Light mode is broken in most screens because colors aren't responsive to theme.

Critical Light Mode is Non-Functional

ThemeProvider exists and supports light/dark/system, but only ~6 of 103 files actually check isDark. Switching to light mode shows white text on white backgrounds in most screens.

High Glassmorphism Underutilized

GlassCard component exists but uses conservative blur (40-50 intensity). Not leveraging iOS-native UIBlurEffect materials (systemUltraThinMaterial, systemChromeMaterial). Tab bar and headers should use native blur, not opaque backgrounds.

High Animations are Basic

Only FadeInDown + spring on mount. Missing: gesture-driven animations, shared element transitions, scroll-based parallax, layout animations for list changes, micro-interactions on touch.

Medium AppBackground Not Used

A beautiful AppBackground component exists with gradient + copper glow + floating particles. But every screen renders its own LinearGradient instead, duplicating 3 lines per screen and missing the particles.

Good Strong Foundation

Well-structured Expo Router navigation, React Query hooks, shared service layer, haptic feedback on interactions, accessibility labels on key elements, and proper use of useReducedMotion in animations.

Theme System

Light Mode is Broken

The app has a working ThemeProvider (system/dark/light + AsyncStorage persistence) and an Appearance screen. But only 6 of 103 files respond to theme changes. Light mode shows cream/white text on white backgrounds.

ThemeProvider exists but is ignored

Critical
ThemeProvider at components/providers/ThemeProvider.tsx correctly resolves system/dark/light and exposes useTheme() with isDark.

Only these files check isDark: AppBackground, GlassCard, AnimatedTabBar, TabLayout, Appearance, MedicalDisclaimer.

Every other file uses hardcoded dark-mode hex values like backgroundColor: '#12100e' and color: '#faf6f0'. In light mode, you get white text on a light background = unreadable.

Every screen duplicates its own background gradient

High
All 28 screens contain this identical pattern: <LinearGradient colors={['#12100e', '#1a1714', '#12100e']} style={StyleSheet.absoluteFill} />

AppBackground component at components/shared/AppBackground.tsx already handles dark/light gradients + floating particles, but is used in only the root layout — not individual screens.
Fix: Remove all per-screen LinearGradients. Use AppBackground in the root layout only. Make screens transparent so the root background shows through (already partially done in TabLayout with sceneStyle: { backgroundColor: 'transparent' }).

NativeWind/Tailwind classes not used in screens

Medium
The Tailwind config defines all 13 semantic colors (bg, surface, card, copper, gold, sage, etc.) and NativeWind is installed. But screens use StyleSheet.create() with hardcoded hex values instead of className="bg-bg text-white". The Tailwind tokens are effectively unused.

Create a centralized colors module

Recommended
Create lib/colors.ts that exports a function getColors(isDark: boolean) returning all semantic tokens.

Dark palette: Current hex values (already polished).
Light palette: Soft pastels matching the AppBackground gradient — bg: '#f5f7fa', surface: '#ffffff', card: 'rgba(255,255,255,0.7)', text: '#1a1714', muted: '#6b7280'.
Pattern: Create a useColors() hook that calls useTheme() internally. Every component imports useColors() instead of hardcoding hex values. This is a single abstraction that fixes all 1,494 hardcoded colors.

iOS-native semantic colors

Recommended
For light mode, use Apple's semantic system colors where appropriate: PlatformColor('systemBackground'), PlatformColor('label'), PlatformColor('secondaryLabel'). These automatically adapt to dark mode, dynamic type, and accessibility settings.

Glassmorphism

Below iOS Standards

Apple's native apps use vibrancy materials extensively (Control Center, Notification Center, Spotlight). The app's GlassCard is a start but doesn't match the level of sophistication expected on iOS.

GlassCard uses conservative blur intensity

High
Current: intensity={isDark ? 40 : 50} with tint="dark".

iOS system sheets use intensity 80-100 with systemUltraThinMaterial or systemChromeMaterial. The current settings produce a barely visible blur effect that feels more like a semi-transparent card than genuine glass.
Fix: Increase intensity to 60-80 for dark mode and 70-90 for light mode. Use expo-blur's tint="systemChromeMaterialDark" for iOS 16+ to get native material appearance. Add a subtle inner border glow: borderColor: 'rgba(255,255,255,0.12)'.

Tab bar and header should use system blur materials

High
Tab bar (TabIndicator.tsx): Uses BlurView intensity={80} — good, but the header at the top uses opaque #12100e background.

Screen headers: All screen layouts use opaque dark backgrounds. Content scrolling behind the header should be visible through a blur, matching Apple's native UINavigationBar behavior.
Fix: Replace opaque header backgrounds with BlurView using tint="systemChromeMaterialDark". Make the content extend behind the header with contentInsetAdjustmentBehavior.

Modal backgrounds use opaque overlays

Medium
Budget screen's custom item modal uses backgroundColor: 'rgba(0,0,0,0.6)' for the overlay and #201c18 for the sheet. No blur effect on the background content.

iOS system modals (Action Sheets, Share Sheets) use background blur to maintain context of what's behind.
Fix: Use BlurView as the modal backdrop. For the bottom sheet, use @gorhom/bottom-sheet with backdropComponent using blur. This matches iOS system sheets and keeps context visible.

Android glass fallback is just opaque

Low
GlassCard falls back to backgroundColor: 'rgba(32,28,24,0.92)' on Android. This is functionally opaque. On modern Android (12+), blur effects are supported.
Consider: For Android 12+, use react-native-blur or custom RenderEffect. For older Android, the current opaque fallback with elevated shadow is acceptable.

The app has 9 animation components using react-native-reanimated. The foundation is solid (spring physics, reduced motion support), but the motion vocabulary is limited to mount-only fade-ins.

Positive: All animation components check useReducedMotion() and disable animations accordingly. This is excellent accessibility practice.

Scroll-Based Parallax

Dashboard header, briefing hero, landing page hero should have parallax effects as user scrolls. Use useAnimatedScrollHandler from reanimated to drive opacity/scale/translation based on scroll offset.

Shared Element Transitions

Navigation between list and detail screens (tasks, briefing weeks, articles) should have connected animations. Use react-native-shared-element or Reanimated 3's shared transitions.

Layout Animations

When tasks are completed/snoozed, they should animate out of the list. When filters change, items should reflow. Use LayoutAnimation.configureNext() or Reanimated's Exiting prop.

Gesture-Driven Cards

Dashboard cards should be draggable for reordering. Pull-down-to-refresh should have a custom copper-themed animation instead of the default spinner. Briefing sections could collapse/expand with gesture.

Interactive Press Feedback

Cards should scale down slightly on press (withSpring(0.97)) like native iOS buttons. Currently only landing page CTAs have pressed && { transform: [{ scale: 0.97 }] }.

Skeleton Loading States

A Skeleton component exists in ui/Skeleton.tsx but screens show a plain ActivityIndicator. Shimmer skeletons that match the card layout would feel more polished.

Apple Health

Scroll-based card stacking, rubber-band overscroll, fluid transitions between summary and detail views. The tracker screen should feel this natural.

Things 3

Buttery smooth task completion (checkmark fills, item slides out with spring). Drag-to-reorder with haptic ticks. The Tasks screen should aspire to this level.

Apple Weather

Background gradients that shift with time-of-day. Parallax layers. The briefing hero section could use similar atmospheric depth.

iOS Notification Center

Grouped notifications with spring-loaded expand/collapse. The notification inbox screen should mirror this pattern.

Implementation Roadmap

Recommended Fix Phases

Ordered by impact and dependency. Each phase is self-contained and shippable.

Phase 1 — Foundation (Highest Impact)

Centralized Color Tokens + useColors() Hook

Create a single source of truth for all colors with dark/light variants. Every subsequent phase depends on this.

Create lib/colors.ts with darkColors and lightColors objects (33 semantic tokens each)
Create hooks/use-colors.ts that reads from ThemeProvider and returns the active palette
Migrate top 15 files (55% of hardcoded colors) to use useColors()
Replace all per-screen LinearGradients with transparent backgrounds (AppBackground handles it)
Design light mode palette: soft pastels matching AppBackground's light gradient
Phase 2 — Light Mode Fix

Make Light Mode Actually Work

With tokens in place, systematically update all remaining files to be theme-aware.

Migrate remaining 88 files from hardcoded hex to useColors()
Update GlassCard with light mode border/shadow variants
Update all icon colors to use theme tokens (currently hardcoded hex in JSX)
Test all 28 screens in light mode; fix any remaining contrast issues
Add theme-aware RefreshControl colors, ActivityIndicator colors
Phase 3 — Native Glass

iOS-Native Glassmorphism Upgrade

Leverage Apple's native blur materials for a polished, premium feel.

Increase GlassCard blur intensity to 60-80 (dark) / 70-90 (light)
Use tint="systemChromeMaterialDark" for iOS 16+ in GlassCard
Add BlurView to main header (replace opaque #12100e background)
Add blur backdrop to modal overlays (budget custom item, alerts)
Add subtle inner border highlights to glass surfaces: rgba(255,255,255,0.08-0.15)
Add glass cards to Dashboard (currently plain Views)
Phase 4 — Motion Design

Sophisticated iOS Animations

Upgrade from mount-only fade-ins to gesture-driven, scroll-aware motion.

Add interactive press feedback to all cards: withSpring(0.97) on press in
Add exit animations to task completion: checkmark fill + slide out with spring
Add scroll-based parallax to dashboard greeting + briefing hero
Add LayoutAnimation for list reflows when filters change
Implement shimmer skeleton loading states (replace ActivityIndicator)
Add spring-loaded expand/collapse for grouped task sections
Coordinate haptic ticks with animation keyframes (task complete, mood select)
Phase 5 — Polish

Premium Details & Micro-Interactions

The finishing touches that separate a good app from a great one.

Custom pull-to-refresh animation (copper-themed instead of default spinner)
Shared element transitions for screen navigation (task list → detail)
Animated FAB button (scale + rotate on scroll direction change)
Confetti/celebration animation on milestone completions
Time-of-day background gradient shifts (dawn warmth, night deep)
@gorhom/bottom-sheet for modals instead of RN Modal (native-feeling drag gesture)
Quick Wins

Immediate Impact, Low Effort

Changes that can be made today with minimal risk and maximum visual improvement.

1. Remove per-screen gradients

Delete all 28 instances of <LinearGradient colors={['#12100e',...]}> from screens. AppBackground already renders in root layout with particles. Screens just need backgroundColor: 'transparent'.

~28 files, ~3 lines each = 84 lines removed

2. Increase GlassCard blur intensity

Change intensity ?? (isDark ? 40 : 50) to intensity ?? (isDark ? 65 : 80) in GlassCard.tsx. One-line change, immediate visual upgrade across all glass-using screens.

1 file, 1 line change

3. Add press scale to all cards

Wrap dashboard cards in Pressable with pressed && { transform: [{ scale: 0.98 }] }. Instant native feel. Already done on landing page CTAs.

~10 components, ~5 lines each

4. Add blur to main header

Replace backgroundColor: '#12100e' in tab layout header with BlurView tint="dark" intensity={80}. Already done for tab bar. Consistent chrome throughout.

1 file: (tabs)/_layout.tsx

5. Add GlassCard to Dashboard cards

Dashboard cards are currently plain Views. Wrap them in GlassCard for visual consistency with Tracker, More, and Briefing screens that already use it.

~8 dashboard card components

6. Randomize CardEntrance stagger delays

Current delays are linear (0, 80, 160, 240...) which feels mechanical. Use slight randomization: delay={index * 80 + Math.random() * 40} for organic feel.

Per-screen, no component change needed
