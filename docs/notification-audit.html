Issues Found

The notification system uses Supabase Edge Functions for delivery, pg_cron for scheduling, Expo Push for mobile, VAPID/Web Push for browser, and Resend for email. Frequency caps (1/hr, 3/day, 10/week) prevent notification fatigue. Partner activity bypasses all caps.

Notification Flow

Trigger (cron job or API call) → Edge Function (send-notifications) → Check preferences + quiet hours → Check frequency caps → Persist in-app notification → Send push (web + mobile) → Send email (if applicable) → Log delivery status

Dashboard recap

15 issues found. Sorted by impact. Red = critical gaps. Yellow = should fix. Green = recommendations.

Critical Gaps

1. Partner Activity Never Triggered

The partner-activity edge function exists and is fully implemented, but nothing in the app calls it. No database trigger, no client-side hook, no API route. Partner notifications don't work at all.

Fix: Add HTTP POST to partner-activity function when tasks are completed or baby logs are created. Either via database trigger on family_tasks/baby_logs or from the client-side mutation hooks.

2. Six Notification Types Not Scheduled

These functions exist in send-notifications but have no cron job: milestones, onboarding nudges, mood reminders, re-engagement, push window warnings, onboarding drip emails. They only fire if manually called via API.

Fix: Add cron jobs: milestones (daily 8 AM), onboarding nudges (hourly), mood reminders (daily 6 PM), re-engagement (daily 11 AM), push window warnings (daily 9 AM), onboarding drip (daily 10 AM).

3. Onboarding Nudge Query Bug

The query filters .is("onboarding_completed", true) then tries to send nudges to users who haven't completed onboarding steps. This is contradictory — it should query users who have completed onboarding but haven't performed specific actions yet.

Fix: Review the logic — likely should be .is("onboarding_completed", true) (users who finished setup but haven't engaged with features yet).

Should Fix

4. Timezone Not Used in Quiet Hours

profiles.timezone exists but checkQuietHours() defaults to UTC. A user in PST with quiet hours 10 PM - 7 AM will get notifications at wrong times.

Fix: Join profiles table when fetching preferences and pass timezone to checkQuietHours().

5. "celebration" Type Defined But Unused

Exists in the notification_type CHECK constraint but never implemented. Dead code in schema.

Fix: Implement (e.g. first task complete, first week done) or remove from schema.

6. No Database Triggers for Real-Time Events

Task completion and baby log creation depend on client-side code to call notification functions. If the client doesn't call, no notification fires.

Fix: Add PostgreSQL triggers on family_tasks (status = 'completed') and baby_logs (INSERT) that call pg_net.http_post() to the partner-activity function.

7. No Notification Cleanup / Retention Policy

The notifications table grows indefinitely. No archival, no TTL, no auto-delete of old read notifications.

Fix: Add a cron job that deletes read notifications older than 90 days, or add a retention policy.

8. No Email Bounce/Open Tracking

Resend integration only logs "sent" status. No tracking of bounces, opens, or clicks. notification_delivery_log has a 'bounced' status that's never populated.

Fix: Configure Resend webhooks to update delivery_log with bounce/open events.

9. No Rate Limiting on Direct Notifications

The type='direct' API endpoint can be called with any user_id and notification_type. No validation that the caller is authorized.

Fix: Validate that direct notifications are only sent by service-role callers from known sources.

10. Vault Secrets May Not Be Set

pg_cron jobs use vault.create_secret() for project URL and service role key. If these aren't set in your Supabase vault, cron jobs silently fail.

Fix: Verify vault secrets exist: SELECT * FROM vault.decrypted_secrets WHERE name IN ('supabase_project_url', 'service_role_key');

Recommendations

11. Add Monitoring Dashboard

Track notification delivery success rates, frequency cap hits, and channel health via a Supabase SQL query or Grafana dashboard.

12. Add Push Window Upgrade CTA

When free users hit day 27, the push window warning should deep-link to the upgrade screen with context about losing notifications.

13. Implement Celebration Notifications

Use the existing "celebration" type for positive reinforcement: first task completed, first week of tracking, streak milestones (7-day mood streak).

14. Add A/B Testing for Notification Copy

Re-engagement and onboarding emails have fixed copy. Test different subject lines and CTAs to optimize open/click rates.

15. Add Unsubscribe Token to Emails

Emails include "Manage preferences" link but no one-click unsubscribe token. CAN-SPAM and Apple require easy opt-out.
